类别
1. 1XX 信息响应
   - 100 continue：服务器已接收请求的初步部分，客户端应继续请求。
   - 101 Switching Protocols：服务器同意切换协议，如从Http切换到WebSocket。
2. 2XX 成功
   - 200 OK 请求成功，服务器返回请求的资源或者数据。
   - 201 请求成功并且创建了新的资源，常用于POST请求。
   - 204 No Content 请求成功但服务器不返回任何内容，常用于删除操作。
3. 3XX 重定向
   - 301 Moved Permanently 资源已经永久移动到新的URL，客户端应该使用新的URL进行访问
   - 302 Found 资源临时移动到新的URL，客户端应该继续使用原来的URL
   - 304 Not Modified 资源未修改，客户端可以使用缓存版本
4. 4XX 客户端错误
   - 400 Bad Request 请求无效或者语法错误，服务器无法处理
   - 401 Unauthorized 请求需要身份验证，客户端未提供有效的凭证
   - 403 Forbidden 服务器理解请求但是拒绝执行，通常是因为权限问题
   - 404 Not Found 请求的资源在服务器上找不到
5. 5XX 服务端错误
   - 500 Internal Server Error 服务器内部错误
   - 502 Bad Gateway 服务器作为网关或者是代理，从上游服务器接收到无效响应
   - 503 Service Unavailable 服务器暂时无法处理请求，通常是服务器压力过大宕机或者维护中。

## HTTP 请求包含哪些内容，请求头和请求体有哪些类型？



请求头：包含请求方法 get/post/put/delete， 请求路径、Http协议版本

请求体：包含各种键值对、请求内容、认证信息等。

空行：用于分隔请求头和请求体

响应体：仅在post、put等方法中存在，包含发生给服务器的数据。



请求头类型：

- 通用head：如Cache-Control、Connection等
- 请求head：Host、User-Agent、Accept、Authorization
- 实体head：Content-Type、Content-Length



请求体类型：

- multipart/form-data  提交表单数据
- application/x-www-form-urlencoded 文件上传或者是复杂表单数据
- binary 二进制文件
- row 数据行 JSON、XML、文本、HTML、JS（application/json、application/xml、text/plain、text/html、application/javascript）



## HTTP 中 GET 和 POST 的区别是什么？

1. get可以通过浏览器直接访问，post不行
2. get没有body信息，post有
3. get是获取资源，post可以是获取资源也可以是请求修改服务器资源
4. get参数可见，post参数存放在body中，相对安全
5. get请求url长度有限，post参数可以放在body中，理论上没有限制。
6. get幂等（重复请求不会改变服务器状态），post非幂等（重复请求可能会导致重复创建资源或者重复操作）。
7. post相对get方式数据安全性更有保障

## TCP 的粘包和拆包能说说吗？



### 粘包

- **什么是粘包？**
- 粘包就是在TCP的传输过程中，发送方传输的多个包被接收方合并成了一个大包，导致接收方无法区分消息的边界。

- **为什么会发生这种情况？**
- 因为，在TCP的传输协议中，是面向对象流的传输，传输消息没有边界，发送方的多个包数据可能一次性发送给接收方。

- **那如何解决这种情况？**

1. 固定消息长度，接收方按固定长度拆分数据。
2. 在包与包之间添加数据分隔符号标识 如换行符，让接收方可以拆分数据正常接收。
3. 添加消息头长度字段，告知接收方具体长度为多少，接收方根据长度取对应长度数据。



### 拆包

- **什么是拆包**？
- 拆包是在TCP数据传输过程中，发送方发生的一个数据包，被接收方拆分成多个数据包，造成无法正常获取完整数据。
- **为什么会发生这种情况**？
- 因为TCP的缓存池长度有限以及数据可能超出了最大传输单元（MTU Max Transport Unit）。

- **那如何解决这种情况？**（与粘包一致 反向理解即可）

1. 固定消息长度，接收方按固定长度拆分数据。
2. 在包与包之间添加数据分隔符号标识 如换行符，让接收方可以拆分数据正常接收。
3. 添加消息头长度字段，告知接收方具体长度为多少，接收方根据长度取对应长度数据。





**只有在TCP传输中会发生这种情况。UDP是无边界的数据传输。**





## 说说 TCP 的四次挥手？



#### TCP的四次挥手

- 为什么需要挥手？

因为TCP有三次握手，确认双方身份信息建立了连接通道，所以需要关闭通道，通知双方停止传输和释放资源。

因此就有了四次挥手。

- 第一次：客户端向服务端发起请求SYN，携带SEQ x，关闭客户端传输通道，但还可以接收数据。
- 第二次：服务端收到请求，关闭接收通道，响应ACK，并返回SEQ x+1 （确认的客户端信息）以及SEQ y（服务端信息）。
- 第三次：服务端处理完客户端的请求数据后，通知客户端，关闭发送数据通道。
- 第四次：客户端ACK，关闭接收数据，收到服务端的关闭通知，告知服务端，一定时长后关闭连接。



执行顺序

客户端：关闭发送

服务端：关闭接收

服务端：关闭发送

客户端：关闭接收



## 说说 TCP 拥塞控制的步骤？

1. 慢启动 (Slow Start) 发送方在连接建立初期，缓慢地增加数据发送速率。初始的拥塞窗口(cwnd)通常为一个 MSS(最大报文段大小)，然后在每次收到 ACK后成倍增加cwnd，直到达到慢启动阈值(ssthresh)或检测到网络拥塞。
2. 拥塞避免 (Congestion Avoidance) 当 cwnd 达到 sthresh后，TCP进入拥基避免阶段，拥塞窗口的增长速度从指数变为线性增长，即每个 RT(往返时间)增加一个 MSS，这一阶段旨在避免激烈的拥塞反应，保持网络稳定性。
3. 快速重传 (Fast Retransmit) 发送方在收到三个重复的 ACK后，立即重传被认为丢失的报文段，而无需等待超时。这减少了重传的延迟，迅速应对数据丢失。
4. 快速恢复 (Fast Recovery) 在快速重传后，TCP 不进入慢启动，而是减小 cwnd 到当前的一半，并设置 ssthresh 为当前新的 cwnd 的值，然后开始线性增加 cwnd，以快速恢复到丢包前的传输速率。
